# Пользовательские сценарии: «Запись на звонок»

Сценарии описывают внешнее поведение системы для ролей **владелец календаря** и **гость**. Согласованы с [functional-requirements.md](./functional-requirements.md).

---

## Глоссарий

| Термин | Значение |
|--------|----------|
| **Владелец** | Единственный предустановленный профиль календаря. Работает в админ-разделе без входа в систему. |
| **Гость** | Анонимный пользователь с `guestId` в `localStorage`. |
| **Тип события** | Шаблон встречи: `id`, название, описание, длительность. |
| **Окно записи** | 14 календарных дней: сегодня … сегодня + 13. |
| **Слот** | Интервал `[startTime, endTime)` в рабочем дне для выбранного типа события. |
| **Свободный слот** | Слот без пересечения с **любыми** существующими бронированиями. |

---

## Сценарии владельца

### US-O01. Создание типа события

**Роль:** владелец  
**Цель:** добавить новый вид бронирования для гостей.

**Предусловия:**

- Владелец открывает админ-раздел.
- Типа с `id = "intro-30"` ещё нет.

**Основной поток:**

1. Владелец переходит к форме создания типа события.
2. Заполняет:
   - `id`: `intro-30`
   - `name`: «Знакомство»
   - `description`: «Короткий созвон для знакомства»
   - `durationMinutes`: `30`
3. Отправляет форму.

**Результат:**

- Тип сохранён в системе.
- На гостевой странице появляется карточка «Знакомство» с описанием и длительностью 30 мин.

---

### US-O02. Создание типа с другой длительностью

**Роль:** владелец  
**Цель:** предложить гостям встречи разной длины.

**Предусловия:**

- Уже есть тип на 30 мин.

**Основной поток:**

1. Владелец создаёт второй тип: `id = "deep-dive"`, `durationMinutes = 60`.
2. Оба типа отображаются на странице выбора у гостя.

**Результат:**

- Гость может выбрать любой из типов; сетка слотов строится по длительности выбранного типа.

---

### US-O03. Ошибка при дублировании id типа события

**Роль:** владелец  

**Предусловия:**

- Тип `intro-30` уже существует.

**Основной поток:**

1. Владелец пытается создать тип с тем же `id = "intro-30"`.
2. Система отклоняет запрос.

**Результат:**

- Показана ошибка о неуникальном `id`.
- Существующий тип не изменён.

---

### US-O04. Просмотр предстоящих встреч всех типов

**Роль:** владелец  
**Цель:** видеть единый список будущих бронирований.

**Предусловия:**

- Типы: «Знакомство» (30 мин), «Консультация» (45 мин).
- 2 будущих бронирования разных типов, 1 прошедшее.

**Основной поток:**

1. Владелец открывает страницу предстоящих встреч.
2. Видит 2 записи в одном списке.
3. У каждой строки: дата, время, название типа, `guestId`.

**Результат:**

- Прошедшие встречи не отображаются.
- Записи отсортированы по дате и времени начала.

---

### US-O05. Пустой список встреч

**Роль:** владелец  

**Предусловия:**

- Нет будущих бронирований.

**Основной поток:**

1. Владелец открывает страницу предстоящих встреч.
2. Видит «Нет предстоящих встреч».

**Результат:**

- Пустое состояние отображается корректно.

---

### US-O06. Редактирование типа события

**Роль:** владелец  
**Цель:** обновить описание или длительность существующего типа.

**Предусловия:**

- Тип `intro-30` существует (30 мин, «Знакомство»).

**Основной поток:**

1. Владелец открывает список типов в админке.
2. Переходит к редактированию `intro-30`.
3. Меняет `name` на «Быстрое знакомство» и `durationMinutes` на `45`.
4. Сохраняет изменения.

**Результат:**

- Гостевая карточка показывает новое название и длительность 45 мин.
- Сетка слотов для новых бронирований строится с шагом 45 мин.
- Существующие бронирования сохраняют прежнюю длительность 30 мин.
- `id` типа остаётся `intro-30`.

---

### US-O07. Удаление типа без предстоящих встреч

**Роль:** владелец  
**Цель:** убрать неактуальный тип из каталога.

**Предусловия:**

- Тип `old-format` существует.
- Нет предстоящих бронирований на этот тип (прошедшие могут быть).

**Основной поток:**

1. Владелец удаляет тип `old-format` из админки.
2. Подтверждает удаление.

**Результат:**

- Тип исчезает из гостевого каталога и админ-списка.
- `DELETE /event-types/old-format` → 204.

---

### US-O08. Запрет удаления типа с предстоящими встречами

**Роль:** владелец  

**Предусловия:**

- На тип `intro-30` есть 1 предстоящее бронирование.

**Основной поток:**

1. Владелец пытается удалить `intro-30`.
2. Система отклоняет операцию.

**Результат:**

- Тип остаётся в системе.
- Показано сообщение: «Нельзя удалить тип с предстоящими встречами».
- API → 409.

---

### US-O09. Просмотр и управление списком типов

**Роль:** владелец  

**Предусловия:**

- Создано 3 типа событий.

**Основной поток:**

1. Владелец открывает `/admin/event-types`.
2. Видит таблицу: `id`, название, длительность, действия (редактировать / удалить).

**Результат:**

- Владелец может перейти к созданию, редактированию или удалению любого типа.

---

## Сценарии гостя

### US-G01. Просмотр доступных типов событий

**Роль:** гость  
**Цель:** выбрать подходящий формат встречи.

**Предусловия:**

- Владелец создал 2 типа событий.

**Основной поток:**

1. Гость открывает главную страницу.
2. Видит карточки с названием, описанием и длительностью каждого типа.
3. Выбирает один тип.

**Результат:**

- Гость переходит к календарю выбранного типа.

---

### US-G02. Успешное бронирование

**Роль:** гость  
**Цель:** записаться на свободный слот.

**Предусловия:**

- Выбран тип «Знакомство» (30 мин).
- Завтра слот 10:00–10:30 свободен.
- Завтра входит в 14-дневное окно.

**Основной поток:**

1. Гость выбирает тип «Знакомство».
2. В календаре видит завтрашний день с `freeSlotCount > 0`.
3. Выбирает завтра → слот 10:00 (статус «свободен»).
4. Нажимает «Записаться».
5. Видит подтверждение: завтра, 10:00–10:30, «Знакомство».

**Результат:**

- Создано бронирование с `eventTypeId`, `guestId`, датой и временем.
- Слот 10:00–10:30 недоступен для других гостей и типов.

---

### US-G03. Конфликт с другим типом события

**Роль:** гость  
**Цель:** на одно время нельзя две записи, даже разных типов.

**Предусловия:**

- На дату D забронировано 11:00–11:30 (тип «Знакомство», 30 мин).
- Гость выбирает тип «Консультация» (45 мин).

**Основной поток:**

1. Гость открывает день D.
2. Слот 11:00 отображается как «занят».
3. Слот 11:30 (старт консультации 45 мин → 11:30–12:15) отображается как «занят» (пересекается с 11:00–11:30).
4. Слот 11:30 для 30-мин типа был бы свободен, но для 45-мин — занят из-за пересечения.

**Результат:**

- API отклоняет любое пересекающееся бронирование с кодом 409.

---

### US-G04. Бронирование только в 14-дневном окне

**Роль:** гость  
**Цель:** запись ограничена ближайшими 14 днями.

**Предусловия:**

- Сегодня — 12 июня.

**Основной поток:**

1. Гость открывает календарь.
2. Видит дни с 12 по 25 июня включительно.
3. 26 июня и далее **не отображаются** как доступные для записи.
4. Прямой вызов API с датой 26 июня → 400.

**Результат:**

- Гость не может записаться за пределами окна.

---

### US-G05. Попытка забронировать прошедший слот сегодня

**Роль:** гость  

**Предусловия:**

- Сейчас 14:20.
- На сегодня есть свободные слоты после 14:20.

**Основной поток:**

1. Гость выбирает сегодня.
2. Слоты до 14:15 (для 15-мин типа) или до 14:00 (для 30-мин) — «прошедшие».
3. Выбирает 14:30 и успешно бронирует.

**Результат:**

- Прошедшие слоты не бронируются и не входят в `freeSlotCount`.

---

### US-G06. День без свободных слотов

**Роль:** гость  

**Предусловия:**

- На дату D все слоты выбранного типа пересекаются с существующими бронированиями.

**Основной поток:**

1. В календаре у дня D бейдж **0**.
2. Гость открывает день D.
3. Все слоты — «занят».
4. Кнопка «Записаться» неактивна.

**Результат:**

- Понятное состояние полной занятости.

---

### US-G07. Попытка забронировать занятый слот (гонка)

**Роль:** гость  

**Предусловия:**

- Слот 15:00 свободен в UI у двух гостей.

**Основной поток:**

1. Оба выбирают 15:00 и нажимают «Записаться» почти одновременно.
2. Первый получает 201.
3. Второй получает 409.

**Результат:**

- Второй гость видит «Слот уже занят», список обновляется.

---

### US-G08. Идентификация гостя через localStorage

**Роль:** гость  

**Предусловия:**

- Первый визит в браузере.

**Основной поток:**

1. Приложение генерирует `guestId`, сохраняет в `localStorage`.
2. При бронировании `guestId` отправляется в API.
3. При повторном визите используется тот же `guestId`.

**Результат:**

- Бронирования привязаны к анонимному идентификатору браузера.

---

### US-G09. Пустой каталог типов событий

**Роль:** гость  

**Предусловия:**

- Владелец ещё не создал ни одного типа.

**Основной поток:**

1. Гость открывает главную.
2. Видит «Нет доступных типов событий».

**Результат:**

- Флоу бронирования недоступен до появления типов.

---

### US-G10. Смена типа события в процессе

**Роль:** гость  

**Предусловия:**

- Гость на календаре типа A.

**Основной поток:**

1. Возвращается к списку типов.
2. Выбирает тип B.
3. Календарь перезагружается с сеткой по длительности B.
4. Ранее выбранный день и слот сброшены.

**Результат:**

- Слоты соответствуют новому типу события.

---

### US-G11. Просмотр своих предстоящих записей

**Роль:** гость  
**Цель:** увидеть активные бронирования текущего браузера.

**Предусловия:**

- `guestId` имеет 2 будущие записи и 1 прошедшую.

**Основной поток:**

1. Гость нажимает «Мои записи» на главной.
2. Видит 2 записи: дата, время, название типа, длительность.
3. Записи отсортированы по дате и времени.

**Результат:**

- Прошедшая запись не отображается.
- Данные соответствуют `GET /bookings?guestId=...&upcoming=true`.

---

### US-G12. Пустые «Мои записи»

**Роль:** гость  

**Предусловия:**

- У текущего `guestId` нет предстоящих бронирований.

**Основной поток:**

1. Гость открывает `/my-bookings`.
2. Видит «У вас нет предстоящих записей» и ссылку «Записаться».

**Результат:**

- Пустое состояние с понятным призывом к действию.

---

### US-G13. Повторный визит — записи сохранены

**Роль:** гость  

**Предусловия:**

- Ранее созданы бронирования в этом браузере.

**Основной поток:**

1. Гость закрывает вкладку и возвращается через несколько дней.
2. `guestId` читается из `localStorage`.
3. «Мои записи» показывают актуальные будущие бронирования.

**Результат:**

- История привязана к браузеру, не к аккаунту.

---

### US-G14. Записи не видны с другого устройства

**Роль:** гость  

**Предусловия:**

- На устройстве A есть бронирования.

**Основной поток:**

1. Гость открывает приложение на устройстве B.
2. Генерируется новый `guestId`.
3. «Мои записи» пусты.

**Результат:**

- Без переноса `guestId` история недоступна на новом устройстве.

---

## Сквозные / граничные сценарии

### US-X01. Смежные слоты без конфликта

**Предусловия:**

- Забронировано 12:00–12:30 (тип A).

**Основной поток:**

1. Гость бронирует 12:30–13:00 (тип B) — **успех**.
2. Попытка забронировать 12:15–12:45 — **отклонена**.

**Результат:**

- Смежные интервалы разрешены; перекрывающиеся — нет.

---

### US-X02. Последний слот рабочего дня

**Предусловия:**

- Тип с `durationMinutes = 30`.

**Основной поток:**

1. Последний доступный старт — **18:30** (окончание 19:00).
2. Старт в 18:45 для 30-мин типа отсутствует в сетке.

**Результат:**

- API отклоняет бронирование за пределами 09:00–19:00.

---

### US-X03. Окно 14 дней на границе месяца

**Предусловия:**

- Сегодня 28 июня. Окно: 28 июня — 11 июля.

**Основной поток:**

1. Календарь показывает дни обоих месяцев в пределах окна.
2. 12 июля недоступен.

**Результат:**

- Окно соблюдается независимо от границ месяца.

---

### US-X04. Ошибка сети при бронировании

**Основной поток:**

1. Гость нажимает «Записаться».
2. Сетевой сбой.
3. Показывается ошибка и «Повторить».
4. До ответа 201 бронирование не считается созданным.

**Результат:**

- Нет ложного подтверждения.

---

### US-X05. Владелец видит новое бронирование

**Предусловия:**

- Владелец смотрит пустой список.
- Гость создаёт бронирование.

**Основной поток:**

1. Владелец обновляет страницу предстоящих встреч.
2. Новая запись появляется с корректным типом события.

**Результат:**

- Админ-список консистентен с БД.

## Матрица сценариев и API

| Сценарий | Метод API |
|----------|-----------|
| US-O01, US-O02 | `POST /event-types` |
| US-O03 | `POST /event-types` → 409 |
| US-O04, US-O05 | `GET /bookings/upcoming` |
| US-O06 | `PATCH /event-types/{id}` |
| US-O07 | `DELETE /event-types/{id}` → 204 |
| US-O08 | `DELETE /event-types/{id}` → 409 |
| US-O09 | `GET /event-types` |
| US-G01, US-G09 | `GET /event-types` |
| US-G02, US-G07 | `POST /bookings` |
| US-G03, US-G06 | `GET /days/{date}/slots?eventTypeId=` |
| US-G04, US-G10 | `GET /calendar?eventTypeId=` |
| US-G05, US-X01, US-X02 | Валидация `POST /bookings` |
| US-G08 | `guestId` в теле `POST /bookings` |
| US-G11, US-G12, US-G13 | `GET /bookings?guestId=&upcoming=true` |
