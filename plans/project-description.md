# «Запись на звонок»

Упрощённый сервис бронирования времени по мотивам Cal.com: владелец календаря настраивает типы встреч, гость выбирает свободный слот и записывается на звонок.

Функциональность намеренно ограничена — без регистрации, авторизации, личных кабинетов и интеграций с внешними календарями. Цель — небольшое, но законченное MVP.

Подробные требования, сценарии и API-контракт — в [functional-requirements.md](./functional-requirements.md), [user-scenarios.md](./user-scenarios.md), [api.tsp](./api.tsp).

---

## Роли

| Роль | Описание |
|------|----------|
| **Владелец календаря** | Один заранее заданный профиль. Админ-раздел без входа в систему. |
| **Гость** | Бронирует без аккаунта. Идентифицируется `guestId` (UUID v4) в `localStorage`. |

---

## Владелец календаря

- **CRUD типов событий** — для каждого типа задаёт `id`, название, описание и длительность в минутах. Поле `id` неизменяемо после создания. Удаление возможно только если нет предстоящих бронирований на этот тип.
- **Предстоящие встречи** — единый список будущих бронирований всех типов событий.

---

## Гость

- **Каталог типов** — страница с видами брони: название, описание, длительность.
- **Бронирование** — выбор типа → календарь → свободный слот → подтверждение.
- **Мои записи** — предстоящие бронирования текущего `guestId`.

### Флоу бронирования

1. Гость выбирает тип события на главной.
2. Открывается календарь на **14 дней** (сегодня … сегодня + 13).
3. На каждом дне — количество свободных слотов для выбранного типа.
4. При выборе дня — список слотов со статусом (свободен / занят / прошедший).
5. Гость выбирает свободный слот и нажимает «Записаться».
6. Отправляется запрос на бронирование с `guestId`, `eventTypeId`, датой и временем.

Занятый слот нельзя выбрать. Прошедшие дни и слоты недоступны.

---

## Правила слотов и занятости

**Рабочие часы:** 09:00–19:00. Последний слот заканчивается в 19:00 (например, при 30 мин последний старт — 18:30, при 15 мин — 18:45).

**Сетка слотов** строится по `durationMinutes` выбранного типа события. Слоты генерируются динамически; в БД хранятся только бронирования.

**Окно записи:** гость может записаться только на даты в ближайшие 14 дней от текущей даты.

**Конфликты:** на одно и то же время нельзя создать две записи, **даже если это разные типы событий**. Пересечение любых интервалов на одну дату запрещено.

---

## Идентификация гостя

Авторизация не требуется. При первом визите фронтенд генерирует UUID и сохраняет в `localStorage`. Этот `guestId` используется при бронировании и на странице «Мои записи».

---

## API

Контракт для бэкенда и фронтенда описан в [api.tsp](./api.tsp) (TypeSpec).

Основные группы эндпоинтов:

- `/event-types` — CRUD типов событий
- `/bookings` — создание и просмотр бронирований гостя
- `/bookings/upcoming` — предстоящие встречи владельца
- `/calendar` — 14-дневное окно для типа события
- `/days/{date}/slots` — слоты конкретного дня

---

## Вне scope

- Регистрация, логин, OAuth
- Несколько владельцев
- Внешние календари, уведомления
- Отмена и перенос бронирования гостем
- Настройка рабочих часов через UI
